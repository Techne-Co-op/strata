Roles reference

15 roles — what each one can see and do

Roles are ecological niches. Each one describes a position in the cooperative's life — what the holder can observe, participate in, and change. Roles are time-bounded grants, not permanent states. They are assigned and revoked by directors, logged in the audit trail.

The member journey

How roles progress

Membership is a threshold, not an event. Most people arrive as guests — seeing only what is public. The path to full membership moves through invitation, application, and admission by the board.

Path to patron membership
guest → applicant → patron_member → (board vote) director / officer
Hub operations path
hub_visitor → hub_applicant → hub_member
Cooperative roles · §18.1 bylaws

Eleven roles for cooperative members

Defined in Bylaws Article 18 §1. Each role is a set of permissions implemented as RLS policies. Having the role does not grant application-level privileges — only database-level row visibility.

Guest
guest
The public. Any visitor to the cooperative's digital surfaces without a session. Sees only what the cooperative has declared public: the bylaws, the hub membership agreement, public documents.
Public bylaws No member data No governance docs
Applicant
applicant
Someone who has submitted a membership application and is under consideration. Can read the documents they need to understand before joining: bylaws and member agreements. Cannot see other members or governance history.
Bylaws Member agreement Own record only No member directory
Patron Member
patron_member
A full voting member. Has paid the $100 share, signed the membership agreement, and been admitted by the board. Can see the member directory, read all cooperative agreements (including minutes and motions), and access their own resource records. One vote among equals.
Member directory All agreements Own resources Own event history Own profile
Investor Member
investor_member
A capital contributor with a separate member agreement (Class Four). Same read access as patron members for governance. Capital record access scoped to own holdings. No voting unless co-holder of a Class One share.
Member directory All agreements Own resources No voting (default)
Suspended Member
suspended_member
A member whose membership has been suspended pending a board resolution. Retains access to their own record and their own event history. Cannot access the member directory, governance documents, or other members' data. Voting rights suspended.
Own record only Own events Own resources No directory No governance
Former Member
former_member
A member whose membership has ended. Retains access to their own historical records: the agreements they signed, their own agent row (including archived_at), their own event history. Cannot access current member data. Their capital account is settled on the board's redemption schedule.
Own record Own agreements Own resources Own events No current members
Director
director
An elected board member. Full read access to all records for governance oversight: complete member directory, all agreements, all resource records, all role assignments, and the full audit event log. Can publish new agreements, manage role assignments, and grant/revoke roles.
All records Full event log Publish agreements Manage roles Board governance
Officer
officer
Treasurer, Secretary, or President. Operational scope: can update agreement supersession chains, manage role assignments, and write governance records. Same governance read access as directors. Officers may manage roles within their operational scope — the application layer enforces the scope; the policy permits the write.
All records Update agreements Manage roles Operational scope
Committee Member
committee_member
A member serving on a board committee (Financial Systems, Operations, etc.). Director-level read access plus committee-specific materials. Cannot publish new agreements or manage roles — those require director or officer level.
All agreements All members All resources No role management
Administrator
administrator
An operational role for system administration — onboarding agents, seeding data, running migrations, populating embeddings. Full read and write access to all tables. All writes are logged. This role is not a governance role; it is an operational one. Actions taken under this role are especially visible in the audit trail.
Full read Full write All tables Fully logged
Nou
nou
The cooperative's intelligence agent. The nou role grants scoped access to read agreement sections (for citation), write event rows (nou.query_received, nou.answer_returned), and read the member directory for coordination. Nou inherits the invoking user's permissions — it cannot access records that the user themselves cannot access. The role is a permission ceiling, not a bypass.
Agreement sections Member directory Write events No role management No resource writes
Hub operations roles · EE Grant

Four roles for the physical space

The Hub at 1515 Walnut, Boulder operates under the Ethereum Foundation grant (EE Grant). These four roles govern access to hub-specific tables — co-working seats, events, KPI measurements. They are defined in the EE Grant §2 and managed by the hub operations module (Wave 2).

Hub Visitor
hub_visitor
Anyone visiting the physical space without a membership. Can see public materials and the event calendar. No access to co-working seat assignments or member data.
Public materials Event calendar No seats
Hub Applicant
hub_applicant
Someone who has submitted a hub membership application. Can read their own application record. Cannot see other applicants or current members' data.
Own application Hub agreement No other records
Hub Member
hub_member
A paying co-working member (Class Two or Class Three). Access to co-working seat assignments, event registration, and hub materials. No access to cooperative governance documents or other members' capital data.
Seat assignments Event registration Hub materials No governance
EE Liaison
ee_liaison
The Ethereum Foundation's designated contact for grant compliance. Access to aggregate KPI measurements and impact reports — not individual member data. Views return aggregates only; row-level data is not exposed. Defined in EE Grant §4.
KPI aggregates Impact reports No individual rows No member data
