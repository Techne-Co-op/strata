Access Policies — Row-Level Security

What each role can see
and what it cannot

Every row in every table is governed by a membrane — a set of PostgreSQL Row-Level Security policies derived from the cooperative's bylaws. What you can read is what the community has chosen to make visible to you.

A cell membrane is not a wall — it is a selective passage. Water enters freely; certain proteins pass only in one direction; some signals require a key. The CIS access layer works the same way. Every table starts completely closed. A row not matched by a permissive policy is invisible — as if it does not exist.

Each policy that opens a passage carries a comment naming the bylaw section that authorizes it. This is the cite-as-you-enforce discipline: the database itself is a governance artifact. When the bylaws change, the policy comment becomes a revision signal. The technical and the legal stay coupled — not by convention, but by design.

No row can ever be hard-deleted. Soft deletion via archived_at timestamp is an UPDATE, covered by update policies. The audit stream records every state change. What has been written stays written.

Five Design Principles

  1. Default deny. RLS enabled on every table. No table is open by default.
  2. Cite-as-you-enforce. Every policy comment names the bylaw or PRD section that justifies it.
  3. Helper-function isolation. Role lookups run as SECURITY DEFINER to prevent recursive RLS evaluation.
  4. Negative tests required. Each policy documents its failing case — what should not be permitted.
  5. No hard deletes. Revocation and archiving are always UPDATE operations. DELETE policies: zero.
