Geological strata form in sequence — each layer deposited on the one before it, each containing a record of what lived and what changed. The CIS migrations are the same: an ordered sequence of commitments, each building on the last, each permanent once applied.

A geological layer cannot be unwritten. Once sediment is deposited, it is part of the record — even if later layers cover it. The CIS migrations work the same way. No migration is ever reversed; no schema object is dropped; no row is hard-deleted. Every change that has been applied is part of the permanent record of what the cooperative's information system was, and when it became that way.

The wave progression is not a project plan — it is a phenological calendar, a record of what grows in what season. W0 is the soil preparation: extensions, schema, types, triggers, the bootstrap agents. W1 is the first growth: capital accounts, patronage tracking. Each subsequent wave depends entirely on what came before. Nothing skips.

Invariants

rules that never change
No Reversals
Migrations are never reversed. Schema objects are never dropped. The substrate is permanent.
Append-Only Data
No hard deletes. State changes are UPDATE (setting archived_at or revoked_at). The audit trigger records every change.
Cite-as-You-Enforce
Every RLS policy carries a bylaw citation. Schema and governance stay coupled by design, not convention.
Wave Sequential
Each wave depends entirely on what came before. No wave can be skipped. No migration assumes a future state.
Idempotent Seeds
Seed migrations use ON CONFLICT DO UPDATE. Safe to re-run without corrupting state.
Service Role Only
Migrations run as the Supabase service role, which bypasses RLS. The service key is never exposed to client code.
